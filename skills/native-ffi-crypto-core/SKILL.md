# Skill: Native FFI Crypto Core

## Objective
Move secret-bearing cryptographic operations out of the managed Dart VM
(where the garbage collector can leave stale copies of sensitive data in
memory) into a native Rust module, bound via Dart FFI, with real memory
protections: page locking, guard pages, and explicit zeroization.

## Rules of engagement
- **Preserve the existing public API of `crypto-e2ee-core` exactly.**
  Every caller elsewhere in the app (`vault-unlock-flow`,
  `account-auth-flow`, `item-type-schema` encryption, etc.) must keep
  working unmodified — only the implementation behind those function
  signatures changes from pure Dart to an FFI/JS-interop call into the
  native module. Do not change function names, parameter shapes, or error
  semantics as part of this migration.
- Prefer **Rust** over C for the native module — memory safety by default
  reduces the odds of introducing a new class of bug while fixing the one
  this phase targets.
- **One Rust crate, conditionally compiled — never two separate crypto
  implementations.** Read `docs/FFI_CROSS_PLATFORM_PLAN.md` in full before
  starting. The cryptographic algorithms (Argon2id, AES-256-GCM, SRP/OPAQUE
  math) are pure computation and compile identically for
  `aarch64-apple-ios`, `aarch64-linux-android`, and `wasm32-unknown-
  unknown`. Only the `secure_mem` module (page-locking, guard pages) is
  gated with `#[cfg(unix)]` vs. `#[cfg(target_arch = "wasm32")]` — real
  `mlock()`/guard pages on native, documented no-ops on Wasm. Zeroization
  (via the `zeroize` crate or equivalent) is universal across all three
  targets and must never be weakened for Wasm.
- **Dart-side bindings differ by platform and must be selected via
  conditional export, not runtime branching.** Native builds (iOS/Android/
  desktop) bind via `dart:ffi` to the compiled `.so`/`.xcframework`. Web
  cannot use `dart:ffi` at all — it binds via `dart:js_interop` to the JS
  glue generated by `wasm-bindgen`/`wasm-pack` around the compiled
  `.wasm`. Both implementations sit behind the same abstract Dart
  interface (`native_crypto_core.dart`), selected with `dart.library.io`
  vs. `dart.library.js_interop` conditional exports — everything above
  this layer imports only the shared interface and never branches on
  platform itself.
- Only lock/guard small, fixed-size secrets (a 32-byte Vault Key, an
  Argon2id output) on native platforms — never attempt to `mlock()` an
  entire decrypted vault payload; iOS's Jetsam process killer enforces
  strict wired-memory limits per app.
- Verify current crate/tooling maturity (Argon2id/AES-GCM crates,
  `wasm-bindgen`/`wasm-pack`, and any Dart-Rust bridge codegen tool) at
  implementation time rather than assuming specific package names or
  versions stay current.
- Write tests at the FFI/JS-interop boundary confirming: (1) bit-identical
  output across the Android, iOS, and Wasm builds for the same fixed test
  vectors — this is what actually proves conditional compilation didn't
  silently change cryptographic behavior anywhere outside `secure_mem`,
  and (2) a deliberately triggered out-of-bounds access in a native-only
  test harness is caught by the guard pages (skip/mark this test as
  platform-specific for the Wasm build, since guard pages don't exist
  there).

## Output location
Native module: `native/crypto_core/` (single Rust crate, see
`docs/FFI_CROSS_PLATFORM_PLAN.md` for the internal layout). Dart bindings:
`core/crypto/native_crypto_core.dart` (shared interface),
`core/crypto/native_crypto_core_io.dart` (FFI, native platforms),
`core/crypto/native_crypto_core_web.dart` (JS interop, Web), selected via
`core/crypto/native_crypto_core_selector.dart`'s conditional export. The
existing `core/crypto/` files from `crypto-e2ee-core` should now delegate
to this interface rather than performing the raw cryptographic math
themselves.
